<h3>Головне правило (TL;DR)</h3>
<p>За замовчуванням завжди роби все <code>private</code>. Відкривай доступ (до <code>protected</code> або <code>public</code>) тільки тоді, коли в цьому з'являється реальна архітектурна необхідність. Це вбереже від "спагеті-коду", коли компоненти лізуть у внутрішній стан один одного.</p>

<h3>1. public (Публічний)</h3>
<p>Доступний скрізь. Усередині класу, в класах-спадкоємцях, в екземплярах класу зовні.</p>

<p class="info">У TypeScript, якщо ти не написав модифікатор, властивість або метод автоматично стає <code>public</code>.</p>

<p>Коли використовувати:</p>

<p> - Для формування публічного API твого класу або сервісу. Це ті методи, заради яких клас і створювався.</p>

<p> - Для методів і властивостей, які викликаються з HTML-шаблонів (особливо актуально для компонентів Angular або Vue 2, де шаблон по суті знаходиться "зовні" класу компонента).</p>

<code class="code">
  class UserService {
    // Цей метод потрібен зовнішньому світу, щоб отримати дані
    public getUser(): User { ... }
  }
</code>

<h3>2. private (Приватний)</h3>

<p>Доступний тільки всередині того класу, де він оголошений. Навіть класи-спадкоємці (через <code>extends</code>) не побачать <code>private</code> методи свого батька.</p>

<p>Коли використовувати:</p>

<p> - Для внутрішнього стану (state), який ніхто не повинен змінювати безпосередньо в обхід твоєї логіки.</p>

<p> - Для допоміжних методів-хелперів, які виконують брудну роботу всередині класу, але зовні абсолютно марні.</p>

<p> - Для інжектування залежностей (DI) у конструкторі сервісів, якщо ці залежності потрібні тільки самому сервісу.</p>

<code class="code">
  class UserService {
    // Внутрішній стан, ніхто не повинен чіпати безпосередньо
    private users: User[] = [];

    // Допоміжний метод, марний зовні
    private formatUserData(data: any): User { ... }

    // DI, якщо HttpClient потрібен тільки цьому сервісу
    constructor(private http: HttpClient) { ... }
  }
</code>

<p class="info info--orange">
  Примітка: У сучасному JS/TS також є хардкорний приватний модифікатор <code>#</code> (наприклад, <code>#balance</code>), який реально приховує дані в рантаймі, на відміну від слова <code>private</code>, яке працює тільки на етапі компіляції TypeScript.
</p>

<h3>3. protected (Захищений)</h3>

<p>Доступний усередині самого класу та в усіх його класах-спадкоємцях. Зовні (у екземпляра) доступу немає.</p>

<p>Коли використовувати:</p>

<p> - При проєктуванні базових/абстрактних класів.</p>

<p>
   - Коли у тебе є спільна логіка для групи компонентів (наприклад, базовий клас табличного компонента з пагінацією), і дочірнім компонентам потрібен доступ до методів батька, щоб перевизначити їх або використовувати, але назовні цей функціонал стирчати не повинен.
</p>

<p> - Коли ти хочеш, щоб спадкоємці могли розширювати функціональність базового класу, але не хочеш відкривати ці методи для зовнішнього світу.</p>

<code class="code">
  abstract class BaseTableComponent {
    // Доступно спадкоємцям, але закрито від зовнішнього світу
    protected currentPage: number = 1;

    protected abstract loadData(): void;
  }

  class UsersTableComponent extends BaseTableComponent {
    public nextPage() {
      this.currentPage++; // Маємо доступ до protected властивості
      this.loadData();
    }

    protected loadData(): void {
      // Реалізація завантаження
    }
  }
</code>

<p class="info">
  Важливо: Не варто зловживати <code>protected</code>, перетворюючи його на "майже public". Якщо тобі здається, що багато методів мають бути <code>protected</code>, можливо, варто переглянути архітектуру і виділити ці методи в окремий сервіс або утиліту.
</p>

<h3>Коли використовувати кожен модифікатор доступу?</h3>

<p>
  <code>protected</code> потрібен тільки для того, щоб розмежувати доступ. Щоб дати можливість використовувати методи в спадкоємцях (адже <code>private</code> там не доступні) і закрити доступ до виклику методів зовні (як для <code>public</code>).
  <br />
  Якщо тобі потрібно, щоб метод був доступний і для спадкоємців, і для зовнішнього світу, то він має бути <code>public</code>.
  <br />
  Якщо ж він потрібен тільки для спадкоємців, то <code>protected</code>.
  <br />
  Якщо ж він потрібен тільки всередині класу, то <code>private</code>.
</p>


<h3>Додатково: Static (Статичні властивості та методи)</h3>

<p>
  <code>Public/Private/Protected</code> відповідають на питання: "Хто це бачить?" (Видимість).
</p>

<p>
  <code>Static</code> відповідає на питання: "Кому це належить?" (Область пам'яті: класу чи об'єкту).
</p>

<p>
  <span class="accent">Статичний метод</span> — це метод, який належить самому класу, а не його екземплярам (об'єктам). Якщо звичайні методи викликаються у конкретного об'єкта, створеного через <code>new</code>, то статичні методи викликаються безпосередньо у класу.
</p>

<h4>Основні особливості</h4>

<p>
  <strong>Виклик без створення об'єкта:</strong> Тобі не потрібно писати <code>const obj = new MyClass()</code>, щоб скористатися методом.
</p>

<p><strong>Відсутність доступу до <code>this</code> (екземпляра):</strong> Всередині статичного методу ти не можеш використовувати <code>this</code> для звернення до звичайних полів класу, тому що статичний метод "не знає", який об'єкт його викликав (об'єкта може взагалі не існувати).</p>

<p><strong>Глобальна точка доступу:</strong> Статичні методи часто використовуються для створення допоміжних (утилітарних) функцій, які логічно пов'язані з темою класу.</p>

<code class="code">
  class Calculator {
    // Статичне властивість
    static PI: number = 3.14159;

    // Статичний метод
    static add(a: number, b: number): number {
      return a + b;
    }

    // Звичайний метод (метод екземпляра)
    sayHello(): void {
      console.log("Привіт від калькулятора!");
    }
  }

  // 1. Викликаємо статичний метод безпосередньо у класу
  const sum = Calculator.add(5, 10);
  console.log(sum); // 15
  console.log(Calculator.PI); // 3.14159

  // 2. А ось так викликати не вийде:
  const calc = new Calculator();
  calc.add(5, 10); // Помилка: Property 'add' does not exist on type 'Calculator'.

  // 3. Щоб викликати sayHello, потрібно створити об'єкт
  const myCalc = new Calculator();
  myCalc.sayHello(); // Працює
</code>

<h4>Коли варто використовувати статичні методи?</h4>
<p>
  <strong>Утиліти:</strong> Коли метод виконує чисту дію, не залежну від стану конкретного об'єкта (наприклад, <code>Math.round()</code> в JavaScript — це статичний метод класу <code>Math</code>).
</p>

<p>
  <strong>Фабрики:</strong> Коли потрібно створити об'єкт якимось особливим способом.
</p>

<code class="code">
  class User {
    constructor(public name: string) {}

    static createAdmin(): User {
      return new User("Admin");
    }
  }
  const admin = User.createAdmin();
</code>

<p>
  <strong>Кешування або загальні налаштування:</strong> Якщо потрібно зберігати дані, спільні для всіх об'єктів цього типу.
</p>

<h4>Підсумок по статичних методах</h4>
<p>
  Якщо методу для роботи не потрібні дані з <code>this</code> (властивості конкретного об'єкта), його сміливо можна робити статичним. Це економить пам'ять і робить код чистішим.
</p>
