<h3>The core of the problem and why we need the Event Loop</h3>
<p class="info info--blue"><code>JavaScript</code> is a single-threaded language. It has only one main thread of execution
(Call Stack), which means it can only do one thing at a time.</p>

<p>
  If heavy tasks (network requests, timers) were executed synchronously,
  they would completely block the page. To prevent this, the
  browser provides its own tools (Web APIs) for background work, and the
  Event Loop acts as a manager that coordinates passing the results of these
  background tasks back to the main JS thread.
</p>

<p>To understand the loop, you need to know the core concepts:</p>

<h3>Call Stack</h3>
<p>
  <strong>Call Stack</strong> is the place where code is currently executing.
  <br>
  It stores execution contexts (which function is currently running) and primitive data types
  (numbers, strings, booleans). The stack works very fast and has a fixed size.
  <br>
  It operates on a <b>LIFO (Last In, First Out)</b> principle.
</p>

<p class="info info--orange">
  The Call Stack has the highest and absolute priority.
  <br>
  Since JavaScript is a single-threaded language, the engine can physically perform only one operation at any given time.
  Therefore, as long as there is at least one function in the stack, no other tasks can be executed.
  <br>
  This is a fundamental principle underlying the entire JavaScript asynchronous model.
</p>

<h3>Heap</h3>
<p>
  <strong>Heap</strong> is a large, unstructured region of memory
  used by the JavaScript engine to store complex (reference) data types.
</p>

<h3>Web APIs</h3>
<p>
  <strong>Web APIs (Background workers)</strong> are external tools provided by the environment (the browser).
  <br>
  When they finish their background work (a timer ticks down, data arrives from a server),
  they take your callback function and send it to the queue. At this point, the Web API's work with this task is done.
  They do not store queues inside themselves.
</p>

<h3>Microtasks and Macrotasks (Task Queues)</h3>
<p>
  Queues are simply lists of functions ready to be executed. They sit between the Web APIs and the engine:
</p>

<p>
  <strong>Microtask Queue:</strong> The queue with the highest priority for delayed code.
  <br>
  This is where callbacks from Promises (<code>.then</code>, <code>.catch</code>) and <code>async/await</code> go.
  <br>
  An important detail: Promises are part of the JavaScript language itself, not a Web API,
  so microtasks are primarily generated by the JS engine itself, not the browser.
</p>

<p>
  <strong>Macrotask Queue:</strong> A queue with lower priority.
  <br>

  What goes here: Callbacks from <code>setTimeout</code>, <code>setInterval</code>, I/O events,
  UI event handlers (clicks, scrolls), script loading, etc.
</p>

<h3>Event Loop - sequence of task execution</h3>

<p>
  <strong>Clearing the Call Stack (Synchronous code):</strong> All regular JS code currently running is executed. The call stack must become completely empty.
</p>

<p>
  <strong>Executing ALL Microtasks:</strong> Once the stack is empty, the browser goes to the microtask queue (promises). It executes them non-stop until the queue is completely empty (including those microtasks that were added right during this step).
</p>

<p>
  <strong>Rendering (UI Repaint):</strong> The stack is empty, microtasks are executed. The browser evaluates whether it is time to update the screen (usually this happens ~60 times per second). If needed, it paints new frames.
</p>

<p>
  <strong>Taking ONE Macrotask:</strong> The browser looks into the macrotask queue (timers, clicks, network events). It takes only the first single task from there and throws it into the Call Stack.
</p>

<p>
  <strong>The cycle repeats:</strong> We return to step 1 (the code of this macrotask executes, the stack clears, microtasks are checked again, and so on).
</p>

<p class="info info--blue">
  <strong>Short formula:</strong>
  <br>
  All synchronous code -> ALL microtasks -> Render -> ONE macrotask -> Repeat
</p>
